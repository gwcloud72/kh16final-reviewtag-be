# Review Tag Backend | 화면별 백엔드 역할 및 상세 설명

이 문서는 포인트 시스템의 실제 화면 흐름과, 그 이면에서 작동하는 백엔드의 역할을 스크린샷과 함께 일목요연하게 정리한 자료입니다. 사용자가 직접 보는 영역은 프론트엔드이지만, 각 화면이 정상적으로 작동하기 위해 백엔드에서 어떤 데이터를 다루고 책임지는지 명확하게 매핑하며 풀어썼습니다.

## 읽는 순서

1. 출석 및 일일 퀘스트 적립
2. 상점 구매 및 인벤토리 사용
3. 포인트 이력 확인
4. 관리자 운영 화면

---

## 01. 출석

![출석 화면](01_attendance.png)

### 화면 설명
매일 한 번, 출석 체크를 할 수 있는 화면이며, 사용자가 도장으로 출석 여부를 확인할 수 있습니다.

### 이 화면을 위한 백엔드 기능 요약
- 오늘 출석했는지 여부 판별
- 연속 출석 일수 계산
- 출석 기록 저장
- 포인트 지급 내역 기록

### 연결 흐름
`AttendanceRestController` → `AttendanceService.checkAttendance()` → `PointService.addPoint()`

---

## 02. 일일 퀘스트 / 퀴즈

![일일 퀘스트](02_quest.png)

### 화면 설명
금일 참여 가능한 퀘스트 목록과 각 퀘스트의 보상 지급 상태를 한눈에 보여줍니다.

### 이 화면을 위한 백엔드 기능 요약
- 해당 일자의 퀘스트 진행 로그 조회
- 퀘스트 진행도 계산
- 보상 중복 수령 방지
- 정답 제출 시, 검증 및 보상 지급

### 연결 흐름
`DailyQuestRestController` → `DailyQuestService` → `POINT_GET_QUEST_LOG` / `PointService.addPoint()`

---

## 03. 상점

![상점 화면](03_store.png)

### 화면 설명
사용자가 포인트로 상품을 구입할 수 있는 공간입니다. 보유 포인트 내역을 확인하고, 원하는 상품을 선택할 수 있습니다.

### 이 화면을 위한 백엔드 기능 요약
- 구매하려는 상품의 존재 여부 확인
- 재고 파악
- 포인트 차감 처리
- 인벤토리 지급 및 업데이트
- 위시리스트 변경

### 연결 흐름
`PointStoreRestController.buy()` → `PointService.purchaseItem()`

---

## 04. 인벤토리

![인벤토리 화면](04_inventory.png)

### 화면 설명  
사용자가 보유한 아이템을 실제로 활용하는 장면을 보여주는 화면입니다.

### 이 화면을 위해 개발한 백엔드 기능  
- 아이템 타입에 따라 사용 방식이 달라지도록 로직을 분리했습니다.  
- 장착형 아이템은 오직 한 가지만 활성화할 수 있도록 구현했습니다.  
- 환불과 삭제와 같은 예외 상황도 처리할 수 있게 만들었습니다.  
- 랜덤 아이템과 닉네임 변경권 등 특수 아이템도 별도의 로직으로 지원하고 있습니다.

### 연결 흐름  
`PointStoreRestController.useItem()` → `PointService.useItem()`

---

## 05. 포인트 이력

![포인트 이력](05_history.png)

### 화면 설명  
어떤 경로로 포인트가 적립되거나 차감됐는지 한눈에 확인할 수 있는 화면입니다.

### 이 화면을 위해 개발한 백엔드 기능  
- 출석, 퀘스트, 구매, 환불, 관리자 지급 등 다양한 상황에서 발생하는 포인트 내역을 모두 한 테이블에 기록합니다.  
- 금액과 적용 사유가 항상 함께 저장되도록 네이밍과 저장 규칙을 통일했습니다.

### 연결 흐름  
`PointService.addPoint()` → `point_history`

---

## 06. 관리자 화면

![관리자 화면](06_admin.png)

### 화면 설명  
운영자가 상품과 포인트 내역을 손쉽게 관리할 수 있는 화면입니다.

### 이 화면을 위해 개발한 백엔드 기능  
- 관리자 권한으로 포인트를 직접 지급하거나 회수할 수 있습니다.  
- 새로운 상품 등록이나 기존 상품 수정, 삭제 기능이 포함되어 있습니다.  
- 회원의 자산 현황을 한눈에 조회할 수 있도록 했습니다.  
- 운영 과정에서도 사용자의 포인트 변동 기준과 동일한 처리가 적용되도록 설계했습니다.

### 연결 흐름  
`AdminRestController` → `PointService.adminUpdatePoint()` / 상품 관리 DAO

---

## 요약

포인트 시스템은 겉보기엔 다양한 화면으로 구성돼 있어 복잡하게 느껴질 수 있지만, 핵심은 결국 **모든 화면이 동일한 잔액과 변동 이력 정보를 바탕으로 움직인다**는 점입니다. 이 문서는 여러 화면에서 시작된 포인트 흐름이 실제로 백엔드에서는 어떻게 연결되고 관리되는지, 그 전체 구조를 한눈에 파악할 수 있도록 간추려 정리한 자료입니다.
