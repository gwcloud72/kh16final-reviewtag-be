# Review Tag | Backend

Review Tag는 **영화·콘텐츠 기반 리뷰 플랫폼**입니다.  
리뷰·별점·가격 평가, 신뢰도 시스템, 자유게시판, 퀴즈, 포인트 기능을 한 서비스 안에 담아 사용자의 참여가 자연스럽게 이어지도록 구성했습니다.

이 저장소와 문서는 프로젝트 전체 기능을 모두 나열하기보다, **팀 프로젝트에서 제가 전담했던 포인트 생태계 영역**을 중심으로 정리했습니다.  
포인트를 단순 보상 수치로 두지 않고, **출석 → 퀘스트 → 적립 → 소비 → 인벤토리/프로필 변화**로 이어지는 흐름을 백엔드에서 설계하고 구현했습니다.

---

## 프로젝트 소개

- 영화·콘텐츠 기반 회원제 리뷰 플랫폼
- 리뷰, 별점, 가격 평가를 함께 제공하는 참여형 서비스
- 신뢰도 시스템과 커뮤니티 기능을 통한 사용자 상호작용 강화
- 포인트를 중심으로 재방문과 활동을 유도하는 보상 구조 설계

---

## 담당 역할

팀 프로젝트에서 **포인트 생태계 구축을 전담**했습니다.

- 출석체크 및 연속 출석 보상 처리
- 일일 퀘스트, 데일리 퀴즈, 룰렛 보상 흐름 구현
- 포인트 상점, 위시리스트, 선물하기 기능 구현
- 인벤토리 사용, 장착, 해제, 환불 로직 구현
- 포인트 이력 관리 및 관리자 운영 기능 구현

프로젝트 전체는 리뷰 플랫폼이지만, 이 문서에서는 **제가 맡은 포인트 도메인에서 어떤 기준으로 설계했고 어떻게 무결성을 맞췄는지**에 초점을 맞췄습니다.

## 핵심 구현 포인트

### 1. 포인트 증감 로직을 한 곳으로 모았습니다.
포인트는 출석, 퀘스트, 구매, 환불처럼 여러 기능에서 공통으로 사용되기 때문에, 증감 로직이 흩어지면 금방 정합성이 깨질 수 있다고 판단했습니다.  
그래서 포인트 지급과 차감은 `PointService`를 단일 진입점으로 두고, **잔액 검증과 이력 저장이 항상 같이 일어나도록** 구성했습니다.

### 2. 적립과 소비가 하나의 흐름으로 연결되도록 설계했습니다.
출석과 퀘스트로 적립한 포인트가 상점, 인벤토리, 장착 기능으로 이어지도록 만들었습니다.  
단순히 포인트만 쌓이는 구조가 아니라, **사용자의 행동이 실제 변화로 보이도록** 설계한 점이 이 도메인의 핵심이었습니다.

### 3. 중복 수령과 중복 처리 문제를 먼저 막았습니다.
출석 중복 체크, 퀘스트 보상 중복 수령 방지, 장착 아이템 단일 활성화, 인벤토리 권한 확인처럼 실제 서비스에서 오류가 나기 쉬운 부분을 우선적으로 막았습니다.

### 4. 운영 측면도 함께 고려했습니다.
사용자 기능만 만드는 데서 끝내지 않고, 관리자 화면에서 포인트 지급·회수, 퀴즈 관리, 상점 상품 관리, 자산 조회가 가능하도록 운영 기능도 같이 구성했습니다.
